
بناءً على المعطيات والمخططات المرفقة الخاصة بتطبيق "فني" (Fanni)، تم إعداد تحليل دقيق للمنطق البرمجي والبنية التقنية المقترحة، مع رصد الثغرات المنطقية والفنية ومقارنتها بأفضل الممارسات العالمية والمحلية في السوق المصري.
يليه مقترح متكامل ومنظم للهيكلة، وقاعدة البيانات، وخطة التنفيذ.

أولاً: تحليل الثغرات المنطقية والفنية (Vulnerability & Gap Analysis)
1. آلية توسيع النطاق الجغرافي للطلبات (Dynamic Radius Expansion)
* الثغرة المنطقية/الفنية: إذا تم تصميم توسيع دائرة البحث (من 5 كم فما فوق) عبر وظائف دورية في الخلفية (Cron Jobs) تقوم بتحديث حقول الطلبات كل بضع دقائق، فسيؤدي ذلك إلى استهلاك ضخم لموارد الخادم وقاعدة البيانات مع زيادة عدد الطلبات النشطة.
* أفضل ممارسة: الاستغناء عن تحديث قاعدة البيانات دورياً، والاعتماد على حساب المسافة "ديناميكياً" في استعلام القراءة (Read Query) الخاص بواجهة الفني. يمكن إدراج عمر الطلب بالدقائق كمتغير داخل استعلام PostGIS لحساب القطر المسموح به في تلك اللحظة.
o معادلة الاستعلام الديناميكي الموصى بها:

2. تتبع موقع الفني في الخلفية واستهلاك البطارية (Background Location Tracking)
* الثغرة الفنية: عند اختيار الفني لـ "الموقع الحالي"، فإن إبقاء نظام تحديد المواقع (GPS) نشطاً في الخلفية عبر Expo Location يواجه قيوداً صارمة من أنظمة التشغيل (iOS/Android)، ويؤدي إلى استهلاك سريع لبطارية الهاتف، مما قد يدفع الفنيين لإغلاق التطبيق.
* أفضل ممارسة: تجنب التتبع المستمر عالي الدقة في الخلفية إلا أثناء رحلة الفني الفعلية إلى العميل (Active Order Delivery). في الحالات العادية، يفضل الاعتماد على "نظام النبضات" (Heartbeat API) الذي يرسل إحداثيات الموقع الحالي على فترات متباعدة (كل 5 إلى 10 دقائق) أو عند تحرك الفني مسافة لا تقل عن 500 متر (Significant Location Changes).
3. مشاكل جودة بيانات العناوين في مصر وخدمة (Nominatim)
* الثغرة الفنية: يعتمد التطبيق على Nominatim (OpenStreetMap) لربط الإحداثيات بالعناوين. على الرغم من تميز جدول location_aliases لحل مشكلة المترادفات العامية، إلا أن خرائط OpenStreetMap تفتقر إلى التغطية الدقيقة لأسماء الشوارع الفرعية والعناوين التفصيلية في الإسكندرية ومصر عموماً مقارنة بخرائط Google. علاوة على قيود معدل الطلبات (Rate Limits) لخدمة Nominatim المجانية.
* أفضل ممارسة: استخدام Nominatim كخيار مجاني ثانوي (Fallback)، والاعتماد بشكل أساسي على Google Places API أو Mapbox للحصول على دقة عناوين عالية في مصر، مع تفعيل الذاكرة المؤقتة (Caching) التي صممتها لتقليل تكلفة الاستهلاك.
4. تزامن معالجة الطلبات ومنع التعارض الجغرافي (Race Conditions)
* الثغرة المنطقية: عند إرسال إشعار لعدة فنيين في نفس النطاق الجغرافي، قد يقوم أكثر من فني بالضغط على "قبول الطلب" في نفس الأجزاء من الثانية. إذا لم تكن قاعدة البيانات محمية، قد تتم عملية القبول المزدوج وتحدث فوضى برمجية وتنظيمية.
* أفضل ممارسة: استخدام ميزة "القفل المتفائل أو المتشائم" (Pessimistic Locking) على مستوى قاعدة البيانات عند معالجة طلب القبول [1]. في PostgreSQL باستخدام Drizzle ORM، يتم تنفيذ الاستعلام باستخدام صياغة تشبه SELECT ... FOR UPDATE لضمان حجز السجل للفني الأول ومعالجة الطلبات الأخرى بالرفض الفوري مع إظهار رسالة مناسبة.
5. أمن وخصوصية المستندات المرفوعة (National ID & Licenses Security)
* الثغرة الفنية: يتطلب التسجيل رفع صور الرقم القومي وصور كارنيه مزاولة المهنة. رفع هذه الملفات الحساسة إلى خادم تخزين عام (Public S3 Bucket) يعرض بيانات المستخدمين للاختراق ويسهل تسريب الهويات الشخصية.
* أفضل ممارسة: تخزين وثائق الهوية الشخصية في حاويات تخزين مغلقة تماماً (Private Buckets) وتوليد روابط مؤقتة مشفرة (Presigned URLs) تنتهي صلاحيتها خلال دقائق معدودة، تُمنح فقط لمسؤولي النظام (Admins) أثناء عملية مراجعة وتفعيل حساب الفني.
6. آلية إتاحة الطلبات المجدولة قبل ساعة واحدة (Scheduled Orders Lead Time)
* الثغرة المنطقية: حصر ظهور الطلب للفنيين قبل ساعة واحدة فقط من موعد الزيارة قد يكون غير كافٍ في المدن المصرية المزدحمة كالإسكندرية، حيث قد يستغرق قبول الطلب والتحرك للعميل وقتاً أطول، مما يسبب تأخر الفني عن الموعد المحدد.
* أفضل ممارسة: جعل "مهلة ظهور الطلب" (Lead Time) متغيرة ومرنة بناءً على نوع الخدمة أو الإعدادات العامة للنظام (مثلاً: ساعتان أو ثلاث ساعات قبل الموعد للأعمال المعقدة)، مع إمكانية إتاحة خيار حجز الطلبات المجدولة مسبقاً للفنيين المتميزين (Scheduled Pool) لترتيب جداولهم اليومية.

تعزيز تجربة المستخدم واجهة الهاتف (Frontend UX):
1. معالجة إزاحة لوحة المفاتيح: استخدام مكتبة @gorhom/bottom-sheet أو react-native-keyboard-controller لضمان إزاحة حقول العنونة بسلاسة عند فتح لوحة المفاتيح، مما يمنع حجب الخريطة أو أزرار الحفظ الحيوية.
2. مؤشر حالة الموقع: وضع مؤشر بصري واضح بلون تفاعلي (أخضر/رمادي) بجانب حقل الإحداثيات يوضح للمستخدم ما إذا كان الموقع الجغرافي قد تم التقاطه بدقة وصحة أم أنه يحتاج إلى إعادة تعيين.

ثالثاً: الهيكلية المقترحة لقاعدة البيانات (Database Schema Optimization)
باستخدام Drizzle ORM و PostGIS، نقترح تصميم الجداول والعلاقات بشكل يضمن كفاءة الاستعلامات الجغرافية والمنطقية:
codeTypeScript
import { pgTable, uuid, varchar, text, integer, timestamp, boolean, customType, index, foreignKey } from 'drizzle-orm/pg-core';

// تعريف نوع مخصص لبيانات PostGIS الجغرافية
const geographyPoint = customType<{ data: string; driverData: string }>({
  dataType() {
    return 'geography(POINT, 4326)';
  },
});

// 1. جدول المستخدمين الأساسي
export const users = pgTable('users', {
  id: uuid('id').primaryKey().defaultRandom(),
  fullName: varchar('full_name', { length: 255 }).notNull(),
  phoneNumber: varchar('phone_number', { length: 20 }).unique().notNull(),
  email: varchar('email', { length: 255 }).unique(),
  role: varchar('role', { length: 50 }).notNull(), // 'client' | 'technician' | 'admin' | 'operator'
  isPhoneVerified: boolean('is_phone_verified').default(false),
  createdAt: timestamp('created_at').defaultNow(),
});

// 2. جدول الفنيين وتفاصيلهم المهنية
export const technicians = pgTable('technicians', {
  id: uuid('id').primaryKey().references(() => users.id, { onDelete: 'cascade' }),
  age: integer('age'),
  nationalId: varchar('national_id', { length: 14 }).unique().notNull(),
  nationalIdFrontUrl: text('national_id_front_url').notNull(), // رابط تخزين خاص وآمن
  nationalIdBackUrl: text('national_id_back_url').notNull(),
  professionId: uuid('profession_id').notNull(),
  subSpecialty: text('sub_specialty'),
  yearsOfExperience: integer('years_of_experience'),
  licenseCardUrl: text('license_card_url'),
  isApproved: boolean('is_approved').default(false), // الموافقة من قبل الإدارة لتفعيل الحساب
});

// 3. جدول العناوين الموحد
export const addresses = pgTable('addresses', {
  id: uuid('id').primaryKey().defaultRandom(),
  userId: uuid('user_id').references(() => users.id, { onDelete: 'cascade' }),
  governorate: varchar('governorate', { length: 100 }).notNull(),
  district: varchar('district', { length: 100 }).notNull(),
  streetName: varchar('street_name', { length: 255 }).notNull(),
  buildingNumber: varchar('building_number', { length: 50 }).notNull(),
  floorNumber: varchar('floor_number', { length: 50 }),
  flatNumber: varchar('flat_number', { length: 50 }),
  coordinates: geographyPoint('coordinates').notNull(), // حقل PostGIS الجغرافي
  isDefault: boolean('is_default').default(true),
}, (table) => ({
  // إنشاء فهرس جغرافي لضمان سرعة الاستعلامات المكانية
  geoIdx: index('geo_index').using('gist', table.coordinates),
}));

// 4. جدول تتبع مواقع الفنيين ومستويات الإتاحة
export const technicianLocations = pgTable('technician_locations', {
  id: uuid('id').primaryKey().defaultRandom(),
  technicianId: uuid('technician_id').references(() => technicians.id, { onDelete: 'cascade' }),
  currentLocation: geographyPoint('current_location'),
  lastWorkLocation: geographyPoint('last_work_location'),
  isAvailable: boolean('is_available').default(false),
  updatedAt: timestamp('updated_at').defaultNow(),
}, (table) => ({
  techGeoIdx: index('tech_geo_index').using('gist', table.currentLocation),
}));

// 5. جدول مرادفات المناطق وعناوين الشوارع باللهجة المحلية
export const locationAliases = pgTable('location_aliases', {
  id: uuid('id').primaryKey().defaultRandom(),
  officialName: varchar('official_name', { length: 255 }).notNull(), // الاسم الرسمي المسجل بقاعدة البيانات
  aliasName: varchar('alias_name', { length: 255 }).notNull(), // المرادف باللهجة العامية أو لغات أخرى
  governorate: varchar('governorate', { length: 100 }).notNull(),
});

تفصيل مهام المرحلة الرابعة والأخيرة (التجهيز للإطلاق الفعلي):
* تأمين ملفات الهوية: ربط واجهات التحميل بخدمات سحابية آمنة تضمن تشفير الوثائق الشخصية المرفوعة للفنيين، لحماية خصوصية المستخدمين وتوافقاً مع القوانين المحلية لحماية البيانات.
* تحسين الـ OTP المطور: توفير إمكانية التحويل اللحظي للمزودين المحليين (مثل Arabyads أو CEQUENS أو Twilio) مع الحفاظ على وضع الـ "Skip" أثناء التطوير لتسهيل عمليات الاختبار ومراجعة متجر التطبيقات بسلاسة.
